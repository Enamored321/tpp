# Інструкція з резервного копіювання (Backup & DR Procedure)

Даний документ регламентує частоту та процедури створення резервних копій, а також план відновлення після збоїв (Disaster Recovery).

Оскільки проєкт є статичним веб-сайтом і не містить власної розгорнутої СУБД (Stateless архітектура), підхід до резервного копіювання є спрощеним. Головним джерелом правди (Single Source of Truth) виступає Git-репозиторій.

## 1. Політика та частота резервного копіювання

### Резервування початкового коду (Source Code Backup)
* **Частота**: Неперервно (під час кожного комміту та push в гілку main).
* **Сховище**: Віддалений репозиторій на GitHub.
* **Правила ротації**: Історія Git зберігається постійно без строку давності, гарантуючи версіювання всього коду.

### Резервування конфігурацій сервера (Infrastructure & Web Server Configs)
* **Що резервувати**: Налаштування (файли з директорії `/etc/nginx/sites-available/` та SSL сертифікати).
* **Частота**: Виконується індивідуально адміністратором після кожної зміни конфігурації.
* **Сховище**: Окремий захищений репозиторій або локальне шифроване сховище (Infrastructure-as-Code).

### Ротація релізних збірок (Artifacts Storage Retention)
* Якщо використовується CI/CD: Кожна успішна збірка (`artifact dist/`) GitHub Actions автоматично зберігається на сервері GitHub з терміном життя 90 днів.
* При ручному оновленні: Локальна копія попередньої стабільної збірки зберігається на вебсервері з ярликом поточної дати (див. `update.md`) і підлягає видаленню через 7 днів для економії дискового простору.

## 2. Відновлення після відмови (Disaster Recovery / DR Phase)

Сценарій: "Повний вихід із ладу сервера. Потрібно підняти сайт з нуля на резервній машині".

### Покрокова процедура відновлення (RTO < 15 хвилин)

**Етап 1. Підготовка інфраструктури (1-5 хвилин)**
1. Активуйте нову віртуальну машину (VPS) з ОС Ubuntu 22.04+.
2. Підключіться через SSH та встановіть базове ПЗ:
   ```bash
   sudo apt update && sudo apt install git nginx -y
   ```

**Етап 2. Відновлення розгортання та коду (1-5 хвилин)**
Якщо відновлюєте за допомогою Docker:
```bash
sudo snap install docker
git clone https://github.com/Enamored321/tpp.git
cd tpp
docker build -t tpp-landing .
docker run -d -p 80:80 --name web tpp-landing
```

Без застосування Docker (ручне встановлення):
```bash
# Встановіть Node.js (v20+)
curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt install nodejs -y

# Отримайте код та зробіть збірку
git clone https://github.com/Enamored321/tpp.git /tmp/tpp
cd /tmp/tpp && npm install && npm run build

# Розгорніть файли та призначте права
sudo mkdir -p /var/www/html/tpp
sudo cp -r dist/* /var/www/html/tpp/
sudo chown -R www-data:www-data /var/www/html/tpp/
```

**Етап 3. Відновлення зв'язку (Health Check & DNS)**
1. Оновіть `A-запис` у DNS-провайдера вашого домену, спрямувавши його на IP нового сервера.
2. Відкрийте сторінку в браузері. Переконайтесь у віддачі коду `HTTP 200 OK` через Developer Tools (Console). Сайт повністю відновлено.
