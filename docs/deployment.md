# Інструкція з розгортання в production (Deployment Guide)

Ця документація призначена для Release Engineers, DevOps або системних адміністраторів, які відповідають за розгортання проекту в робочому (production) середовищі. 

Оскільки проект являє собою повністю статичний Landing Page, зібраний за допомогою Vite, процес розгортання є максимально спрощеним.

## 1. Вимоги до апаратного забезпечення
Для стабільної роботи статики не потрібно потужних ресурсів. 
- **Архітектура:** x86_64, ARM64 (будь-яка, що підтримується сучасними веб-серверами).
- **CPU:** Мінімум 1 ядро (1 vCPU).
- **Оперативна пам'ять (RAM):** Мінімум 512 MB.
- **Дисковий простір:** 50 MB для розміщення файлів розгортання плюс накладні витрати на ОС та веб-сервер (рекомендовано від 5 GB, SSD/NVMe).

## 2. Необхідне програмне забезпечення
- **Операційна система:** Будь-яка серверна ОС на базі Linux (рекомендовано Ubuntu 20.04/22.04/24.04 LTS або Debian 11/12).
- **Веб-сервер:** **Nginx** (рекомендовано через високу продуктивність) або **Apache HTTP Server**.
- **(Опціонально)** **Node.js** (v20+), **npm** та **Git**, якщо ви плануєте робити збирання (`npm run build`) безпосередньо на бойовому сервері (а не CI/CD).
- **Certbot** (для генерації та автоматичного оновлення SSL-сертифіката Let's Encrypt).

## 3. Налаштування мережі
Сервер має розрішати вхідні з'єднання на наступні порти:
- **80 (TCP):** HTTP трафік (з переадресацією на HTTPS).
- **443 (TCP):** HTTPS трафік (основний протокол).
- Переконайтеся, що ваш домен вказує на публічну IP-адресу вашого сервера в налаштуваннях DNS (A-запис).

## 4. Конфігурація серверів (на прикладі Nginx)
Нижче наведена базова конфігурація для Nginx. 

Встановіть Nginx, якщо це не зроблено:
```bash
sudo apt update && sudo apt install nginx
```

Створіть файл конфігурації за адресою `/etc/nginx/sites-available/tpp_landing`:
```nginx
server {
    listen 80;
    server_name example.com www.example.com; # замініть на ваш домен

    root /var/www/html/tpp; # шлях до зібраної статики (dist)
    index index.html;

    # Зтиснення для швидшої передачі
    gzip on;
    gzip_types text/plain text/css application/json application/javascript application/x-javascript text/xml application/xml application/xml+rss text/javascript image/svg+xml;

    location / {
        try_files $uri $uri/ /index.html;
        
        # Кешування статичних файлів
        location ~* \.(?:ico|css|js|gif|jpe?g|png|woff2?|eot|ttf|svg|webmanifest)$ {
            expires 6M;
            access_log off;
            add_header Cache-Control "public, max-age=15552000, immutable";
        }
    }
}
```

Увімкніть сайт та перезапустіть Nginx:
```bash
sudo ln -s /etc/nginx/sites-available/tpp_landing /etc/nginx/sites-enabled/
sudo nginx -t
sudo systemctl reload nginx
```

## 5. Налаштування СУБД
**Не потрібно.** Проект є статичним вебсайтом і не потребує бази даних для свого самостійного функціонування.

## 6. Розгортання коду
Найкращою практикою є попереднє складання артефактів через CI/CD-систему (напр. GitHub Actions), проте ось приклад "ручного" розгортання безпосередньо на сервері (або в CI):

```bash
# Клонуємо репозиторій (або оновлюємо існуючий)
git clone https://github.com/Enamored321/tpp.git /tmp/tpp
cd /tmp/tpp

# Встановлюємо залежності та білдимо проект
npm install
npm run build

# Копіюємо вміст зібраної папки dist/ до директорії веб-сервера
sudo mkdir -p /var/www/html/tpp
sudo cp -r dist/* /var/www/html/tpp/

# Робимо Nginx власником папки для безпеки
sudo chown -R www-data:www-data /var/www/html/tpp
```
*Альтернатива: замість власного сервера, проект можна розгорнути в один клік у GitHub Pages або Vercel.*

## 7. Перевірка працездатності
Після розгортання виконайте наступні кроки, щоб переконатися, що все працює правильно:
1. **HTTP/HTTPS доступність:** Відкрийте браузер і перейдіть за адресою вашого домену (або IP-адресі сервера). Сторінка повинна завантажитись миттєво та без помилок.
2. **Перевірка консолі браузера (F12):** Переконайтеся, що на вкладках Console та Network немає помилок 404 (Файли не знайдені) або 500. Всі CSS стилі, JS скрипти та зображення (зокрема `.svg`) повинні віддаватися зі статусом `200 OK`.
3. **Responsive дизайн:** Звузьте вікно браузера, щоб переконатися, що всі елементи продовжують відображатися коректно на "мобільному" екрані.
4. **Перевірка кешування:** Оновіть сторінку (без очищення кешу). У вкладці Network, статичні ресурси мають завантажуватися зі статусом `(disk cache)` або `(memory cache)` і займати <10ms, що означає роботу кешування Nginx і браузера.
