# Дослідження систем логування та обробки помилок

Цей документ створено на виконання вимог щодо впровадження ефективної системи логування та обробки помилок для Frontend-застосунку.

## 1. Дослідження систем логування
Оскільки даний проєкт є клієнтським застосунком (працює безпосередньо у браузері клієнта з використанням Vanilla JavaScript), серверні системи логування (такі як `Winston` чи `Pino` для Node.js або `Log4j` для Java) не можуть бути застосовані безпосередньо.
Для JavaScript у браузері найефективнішими рішеннями є:
1. **Об'єкт `console`**: Вбудований інструмент браузерів (console.log, console.warn, console.error, console.table) - забезпечує базовий вивід, але не має динамічного управління рівнями відображення на Production.
2. **Сторонні бібліотеки логування (наприклад, `loglevel`)**: Дуже легка (1.4kb) бібліотека, яка заміняє `console` і пропонує встановлення рівнів логування.
3. **Хмарні трекери (Sentry, LogRocket, Datadog)**: Використовуються для отримання звіту про критичні помилки прямо до розробників, коли проєкт вже працює у десятків тисяч реальних відвідувачів.

Для нашого проєкту реалізовано **власний легковагий клас `Logger`** (у файлі `js/logger.js`). Він повністю відповідає вимогам завдання, не перевантажує проєкт сторонніми npm-залежностями і має вбудовану систему управління рівнями логів.

## 2. Аналіз типів помилок у проєкті
Оскільки застосунок статичний та містить інтерфейсні скрипти:
1. **DOM Errors (Reference / TypeError):** Збій пошуку HTML-елементів (наприклад, якщо `id` або клас було видалено), що унеможливлює прив'язку подій чи анімацій.
2. **Алгоритмічні помилки:** У разі перенесення модулів IWD на клієнт можливі логічні помилки (поділ на нуль) або перевищення ліміту рекурсії (Stack Overflow) при проходженні великого графа.
3. **Мережеві (Опціонально):** Помилки HTTP/API при спробі завантажити зовнішні ресурси (наприклад CORS або 404 Not Found).

## 3. Визначення та формат рівнів логування
Власна система `Logger` підтримує 5 рівнів. Формат логу: `[час] [РІВЕНЬ] [модуль] Повідомлення`.

- **DEBUG (1)**: Діагностична інформація, корисна розробнику. *Приклад: "Елемент X з'явився у полі зору (IntersectionObserver)".*
- **INFO (2)**: Життєвий цикл застосунку. *Приклад: "Застосунок запущено", "Скрипти ініціалізовано успішно".*
- **WARNING (3)**: Потенційна проблема, що не призводить до збою. *Приклад: "Навігаційне посилання вказує на якір, якого немає в HTML".*
- **ERROR (4)**: Помилка або виключення (`catch`), що обмежує функціонал частини сайту, проте загальна робота сторінки триває.
- **CRITICAL (5)**: Надкритична помилка (Crash) на етапі старту, яка робить додаток неробочим. Може передбачати автоматичну відправку звіту на сервер.

## 4. Визначення рівня логування без перекомпіляції (Вимога на 65%)
Для Frontend розгортання використання файлів оточення `.env` можливе тільки під час збірки (build step). Це означає, що змінити `.env` без перекомпіляції Vite не вийде.

Тому реалізовано гнучкий підхід: динамічне зчитування конфігурації логера із `localStorage` (сховища браузера користувача).

```javascript
const userLevel = localStorage.getItem('APP_LOG_LEVEL');
```

**Як це працює у Production:**
За замовчуванням система виводить помилки та `INFO`. 
Якщо тестувальнику у Production-середовищі потрібно простежити роботу системи (включити `DEBUG`), йому не потрібно мати доступ до файлів збірки чи перекомпілювати сайт. Достатньо відкрити консоль розробника (F12) та ввести команду:
```javascript
localStorage.setItem('APP_LOG_LEVEL', 'DEBUG');
location.reload();
```
Тепер розширене логування працюватиме для цього користувача доти, поки він не видалить цей локальний ключ налаштувань.
